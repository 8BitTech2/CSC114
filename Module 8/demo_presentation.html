c2c gap analyzer — demo walkthrough

Does the curriculum actually
prepare students for the job?

A working pipeline that compares FTCC's Cloud Management AAS against real job requirements — and an agent that explains the gaps in plain language.

01 · the goal

Check readiness. Report the gap.

The agent checks whether FTCC's Cloud Management curriculum actually prepares students for the cloud and data center jobs available locally — and produces a plain-language report on where the gaps are.

02 · the data

Three real sources, no synthetic data.

curriculum

15 real FTCC Cloud Management courses (A25590U), via NCCCS's statewide standardized catalog text — FTCC's own PDFs have a font-encoding bug that corrupts extraction.

jobs

Real O*NET task data for 5 occupations: Network & Computer Systems Administrators, Information Security Analysts, Computer User Support Specialists, Computer Network Architects, Computer Systems Engineers/Architects.

local

Real, currently posted listings from GDIT (Fort Liberty) and county-government IT roles — used to word the scenario tasks and survey items in actual employer language.

03 · the approach

Classify. Score. Explain.

01

9-category skill taxonomy — virtualization, networking, cloud platforms, storage, security, scripting, monitoring, hardware, soft skills.

02

Course and job text weak-labeled by keyword match, then a TF-IDF + OneVsRest LogisticRegression classifier trained on those labels for smoother scoring.

03

Scores normalized against the dataset's own achievable ceiling — a 0–100 readiness score per category, per occupation.

04

An agent step wraps the whole pipeline and calls Claude to turn the numbers into a natural-language gap report with recommendations.

04 · improvement, example a

A rare-class bug in the classifier.

cloud_platforms scored 3.5/100 despite CTI-141 being a dedicated cloud course. First guess — thin keywords — didn't move the number at all.

Real cause: only 1 of 74 training documents was a positive example for this category. Without class balancing, LogisticRegression suppressed its probability to near-zero, even on the true positive.

05d · improvement, example c

Trusting the expert over the automation.

Hand-coded every course against every taxonomy category, then checked it against the automated classifier's own labels — as an actual test of the model, not just another tuning pass.

Result: Cohen's kappa = 0.073 between expert coding and the classifier — slight agreement. 84 of 144 cells disagreed, almost all in one direction: the classifier was under-counting real coverage that the expert could see and the classifier couldn't.

Fix: curriculum-side scoring now comes directly from the expert-coded matrix, not classifier predictions. The classifier still runs, but only on the O*NET job-task side, where no expert coding exists.

06 · the live demo

Running it, now.

run

python3 agent.py — loads real data, trains the classifier, scores the gap analysis, calls Claude for the report.

fallback

If the LLM call can't reach the API live, the script's own fallback produces a templated report from the same real scored data — documented behavior, not a crash.

13 · closing

What's still unresolved.

rater

Expert-vs-classifier agreement is now measured (kappa=0.073), but that's not the same as two independent human raters agreeing — a true second-rater check is still outstanding.

real gap

hardware_physical_infra stayed relatively weak across every fix — only NOS-110 touches it directly. A genuine curriculum gap, not an artifact.
